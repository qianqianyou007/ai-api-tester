# AI 生成 vs 人工设计 — 测试用例对比分析

---

## 一、数据总览

| 维度 | AI 生成 | 人工设计 |
|------|:---:|:---:|
| 总用例数 | 20 | 11 |
| 正常场景（positive） | 5 | 3 |
| 异常场景（negative） | 5 | 4 |
| 边界场景（boundary） | 5 | 2 |
| 鉴权场景（auth） | 5 | 2 |
| 覆盖端点数 | 5 | 7 |
| 覆盖 HTTP 方法 | 仅 GET | GET + POST + DELETE |

---

## 二、AI 擅长的

### 2.1 批量生成效率极高
5 个端点 × 4 种场景 = 20 条用例，DeepSeek 在约 30 秒内全部生成完毕。人工写 20 条至少需要 10-3- 分钟。效率差距约 **20-60 倍**。

### 2.2 模式统一、结构规范
每个端点严格遵循 positive / negative / boundary / auth 四象限，不会遗漏基础场景。人工容易只关注正常场景，忽略边界和鉴权。

### 2.3 优化后状态码准确率达 95%
经过 Few-shot 示例注入 + 验证层自动修正，20 条中 19 条预期状态码与实际一致。只有一个"不存在用户的 repos 应返回 200 空数组还是 404"存在争议。

### 2.4 使用真实测试数据
优化后的 Prompt 包含了 octocat、no-such-user-000 等可用数据，AI 不再编造假用户名。

---

## 三、AI 遗漏的

### 3.1 只覆盖 GET 方法（最大短板）
AI 生成的 20 条全是对公开数据的 GET 请求。但接口测试的核心难点往往是 **POST / PUT / DELETE 等写操作**：

| 测试点 | AI | 人工 |
|--------|:--:|:--:|
| GET 查询 | ✅ 全覆盖 | ✅ |
| POST 创建 Issue | ❌ | ✅ MANUAL-005 |
| POST 缺少必填字段 → 422 | ❌ | ✅ MANUAL-007 |
| DELETE 删除仓库 → 403 | ❌ | ✅ MANUAL-012 |

根因：AI 的 Prompt 和 Few-shot 示例全是 GET 场景，没有给写操作的示范。

### 3.2 参数组合测试缺失
人工设计了 `per_page` 参数的最大值（100）、最小值（1）、非法类型（"abc"）、排序字段（sort=updated）等多维度测试。AI 只给了"空值"一种边界，没有参数组合概念。

### 3.3 鉴权场景过于肤浅
AI 的 5 条 auth 用例全部是"公开接口不带 Token → 返回 200"。这只是验证了"公开接口不需要认证"，但没覆盖：

| 真正的鉴权场景 | AI | 人工 |
|------|:--:|:--:|
| 需认证接口不带 Token → 401 | ❌ | ✅ MANUAL-011 |
| 带 Token 但权限不足 → 403 | ❌ | ✅ MANUAL-012 |

### 3.4 缺少业务链路测试
AI 把每个端点当作孤立单元。人工设计的 POST 创建 Issue + GET 查询 Issue 形成了一条"创建→验证"链路，这是端到端测试的核心思维。

### 3.5 边界场景单一
AI 对"边界"的理解集中在"参数为空"，输出 5 条全是空字符串场景。人工的边界覆盖更丰富：数值边界（per_page=1, 100）、参数类型校验（传入非数字）、必填字段缺失。

## 四、人工补充的亮点

| # | 用例 | AI 为什么没想到 |
|---|------|------|
| MANUAL-005 | POST 新建 Issue | AI 未被引导生成写操作，需在 Prompt 中加入 POST 示例 |
| MANUAL-007 | POST 缺失必填 title → 422 | 参数校验是接口测试核心，但 AI 不知哪些字段是必填的 |
| MANUAL-008 | per_page 传字符串 "abc" → 422 | 参数类型校验是经典边界，AI 的"空值"思维太窄 |
| MANUAL-009 | per_page=100 最大值 | AI 不理解分页参数的取值范围 |
| MANUAL-011 | 需认证接口无 Token → 401 | AI 只测了公开接口，没选需要认证的端点 |
| MANUAL-012 | DELETE 操作权限不足 → 403 | 多种 HTTP 方法 + 权限模型，超出 AI 当前能力 |

## 五、结论

### 5.1 量化对比

| 指标 | AI 生成 | 人工设计 | 差距 |
|------|:--:|:--:|:--:|
| 生成速度 | ~30 秒 / 20 条 | ~1 小时 / 10 条 | AI 快 100× |
| HTTP 方法覆盖 | 1 种（GET） | 3 种（GET/POST/DELETE） | 人工胜 |
| 参数维度 | 1 种（空值） | 4 种（空值/极值/类型/缺失） | 人工胜 |
| 鉴权深度 | 浅（公开接口无 Token） | 深（需认证/权限不足/403） | 人工胜 |
| 业务链路 | 无 | 有（创建→查询） | 人工胜 |
| 结构规范性 | 高（四象限全覆盖） | 中等（场景分布不均） | AI 胜 |

### 5.2 最佳实践

**AI 生成 + 人工补充 = 效率 × 质量**

1. 让 AI 批量生成 GET 类基础场景 → 覆盖 60-70% 的常规用例
2. 人工聚焦 AI 盲区：写操作（POST/PUT/DELETE）、参数组合、权限模型、业务流程
3. 将人工补充的优秀用例写成 Few-shot 示例 → 反哺 AI，下一轮生成质量更高

### 5.3 下一步改进方向

1. 给 Prompt 加入 POST/PUT/DELETE 的 Few-shot 示例
2. 加入参数边界规则（per_page 范围、必填字段校验）
3. 引入"端点依赖链路"概念（如：POST 创建 → GET 查询验证 → DELETE 清理）
